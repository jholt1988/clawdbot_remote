# MetaÔÇĹScheduler ÔÇö Notion Schema Spec

This spec defines **four global databases** used by MetaÔÇĹScheduler:

1. **Projects Global** ÔÇö the canonical list of projects / initiatives.
2. **Tickets Global** ÔÇö the canonical unit of work (tasks, issues, deliverables).
3. **Calendar Global** ÔÇö scheduled blocks (events, focus blocks, meetings, deadlines).
4. **Drift Log** ÔÇö audit trail of plan vs reality (drift, slips, interruptions, reÔÇĹplans).

The guiding principles:

- **Global-first**: no per-project ÔÇťshadow task lists.ÔÇŁ Everything references the same Tickets Global.
- **Stable IDs**: create a humanÔÇĹreadable ID property for each DB; never reuse IDs.
- **Relations are the source of truth**: avoid duplicating data that is implied by relations.
- **Governance**: changes are explicit; drift is recorded, not handÔÇĹwaved.

---

## Shared conventions (applies to all DBs)

### Naming and ID rules
- **Name** (Notion title property) is required.
- Add a stable text ID:
  - Projects: `P-0001`
  - Tickets: `T-000001`
  - Calendar: `C-000001`
  - Drift: `D-000001`
- IDs are generated by automation OR manually using the next available number.
- **Never change an ID** once assigned.

### Time & timezone
- Store times in Notion date properties (which are timezone-aware).
- Use **UTC for automation**, but keep Notion display in the userÔÇÖs local timezone.

### Status vocab (canonical)
Use these exact labels where applicable:
- `Backlog`
- `Planned`
- `In Progress`
- `Blocked`
- `Waiting`
- `Done`
- `Canceled`

### Priority vocab (canonical)
- `P0` (urgent/critical)
- `P1` (high)
- `P2` (normal)
- `P3` (low)

### Tagging
- Tags are for *cross-cutting* categorization (domain, area, context).
- Do not encode status or priority in tags.

### Governance rules (Drift)
- Drift entries should be **append-only**. Edits are allowed only to fix typos or add missing links.
- Create a Drift entry when any of the following occurs:
  - A due date moves materially (>= 1 day)
  - A calendar block is canceled or moved materially (>= 30 min)
  - A ticket estimate is discovered to be off by >50%
  - A ticket is blocked by an unexpected dependency
- Each Drift entry should link at least one of: `Ticket`, `Project`, or `Calendar Item`.

### Permissions (Drift)
- Jordan: full edit
- Collaborators: comment only
- Automations: may create drift entries based on scheduling changes.

# Governance & operating procedures (cross-database)

### Single source of truth
- Work should exist as a **Ticket**. A calendar block without a ticket is treated as overhead; link it or label it clearly.

### Planning cadence
- Weekly: ensure `Planned` tickets have either a calendar block or an explicit plan.
- Daily: schedule focus blocks from `Planned`/`In Progress` tickets.

### Hygiene
- No orphan calendar items: weekly audit `Calendar Global` where `Related Ticket` is empty and `Kind` is Focus/Admin.
- No orphan drift entries: weekly audit `Drift Log` where all relations are empty.
- If a project is Active (Planned/In Progress), it must have at least one Open ticket.

### Change management
- Renaming properties is discouraged because it breaks automations; add new properties and deprecate old ones.
- Any new select options must be added to this spec.
